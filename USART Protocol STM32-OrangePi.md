# Протокол взаимодействия микрокомпьютера Orange Pi 5 Max с микроконтроллером STM32F401CUU6 в рамках системы Дронпорт

## 1. Общие положения
Настоящий документ описывает низкоуровневый протокол обмена данными между одноплатным микрокомпьютером Orange Pi 5 Max (далее — OrangePi) и микроконтроллером STM32F401CUU6 (далее — STM32). Протокол предназначен для управления силовой электроникой дронпорта, сбора телеметрии датчиков и обеспечения безопасной эксплуатации системы. Взаимодействие строится по принципу ведущий-ведомый, где OrangePi выступает оркестратором, инициирующим команды и запросы, а STM32 исполняет их и предоставляет данные с периферии. Связь осуществляется посредством последовательного интерфейса USART без использования подтверждений доставки пакетов (ACK/NACK), надежность обеспечивается контролем целостности данных и повторными запросами на уровне приложения.

## 2. Физический уровень (Physical Layer)
Обмен данными производится через асинхронный последовательный порт UART. Электрические параметры интерфейса должны строго соответствовать следующим значениям:
*   **Скорость обмена (Baudrate):** 115200 бод.
*   **Логические уровни:** 3.3В TTL (совместимы оба устройства).
*   **Линии связи:** TX (передача), RX (прием), GND (общая земля).
*   **Управление потоком:** Аппаратное управление потоком (RTS/CTS) не используется.
*   **Топология:** Соединение типа «точка-точка» (1:1). Адресация устройств не требуется.
*   **Инициализация:** Процедура рукопожатия при включении питания не предусмотрена. Оба устройства считаются готовыми к работе сразу после инициализации периферии при подаче питания.

## 3. Структура пакета данных
Передача данных осуществляется потоком байтов. Каждый логический сообщение инкапсулируется в пакет фиксированной структуры. Границы пакета определяются специальными байтами-маркерами. Порядок байтов в многобайтовых полях данных — Little Endian (младший байт перед старшим).

Структура пакета состоит из следующих полей, передаваемых последовательно:
1.  **BYTE_START (1 байт):** Байт начала пакета. Всегда равен `0xBF` (двоичное `0b10111111`).
2.  **TYPE (1 байт):** Байт типа сообщения. Содержит кодировку источника, типа команды и целевой подсистемы (подробнее в разделе 4).
3.  **LEN (1 байт):** Длина поля данных (DATA). Может принимать значения от 0 до 255. Если данных нет, значение равно 0.
4.  **DATA (0..255 байт):** Полезная нагрузка сообщения. Содержание зависит от поля TYPE.
5.  **CHECKSUM (1 байт):** Байт контрольной суммы (подробнее в разделе 6).
6.  **BYTE_END (1 байт):** Байт конца пакета. Всегда равен `0xFF` (двоичное `0b11111111`).

## 4. Декодирование байта типа (TYPE)
Байт TYPE является битовой маской, определяющей семантику пакета. Биты нумеруются от 0 (младший) до 7 (старший).

*   **Bit 0 (Source):** Источник сообщения.
    *   `0` — сообщение от STM32 (телеметрия, отчет об ошибке).
    *   `1` — сообщение от OrangePi (команда, запрос статуса).
*   **Bit 1-2 (Command Type):** Тип команды.
    *   `00` — Запрос статуса (Request).
    *   `01` — Ответ со статусом (Response).
    *   `10` — Команда на действие (Action).
    *   `11` — Сообщение об ошибке (Error).
*   **Bit 3-5 (System Part):** целевая подсистема или тип данных.
    *   `000` — Программная часть (Software).
    *   `001` — Датчики Холла (Hall Sensors).
    *   `010` — Напряжение аккумулятора (Voltage).
    *   `011` — Датчик температуры и влажности (DHT22).
    *   `100`..`111` — Зарезервировано.
*   **Bit 6-7 (Reserved):** Резервные биты. Всегда должны быть установлены в `0`.

## 5. Спецификация поля данных (DATA)
Содержимое поля DATA зависит от комбинации битов Command Type и System Part в байте TYPE.

### 5.1. Запрос статуса (Command Type: 00)
Поле DATA отсутствует (LEN = 0). OrangePi отправляет запрос для получения текущего состояния конкретной подсистемы.

### 5.2. Ответ со статусом (Command Type: 01)
*   **Датчики Холла (System Part: 001):** DATA длиной 1 байт. Каждый бит с 0 по 6 соответствует состоянию отдельного датчика Холла (0 — не активен, 1 — активен). Бит 7 всегда 0.
*   **Напряжение АКБ (System Part: 010):** DATA длиной 2 байта (uint16_t). Значение представляет напряжение в вольтах, умноженное на 10 (например, 255 = 25.5В). Порядок байтов: Little Endian.
*   **DHT22 (System Part: 011):** DATA длиной 5 байт. Передается сырой пакет данных (raw data) непосредственно с датчика (влажность старший/младший, температура старший/младший, контрольная сумма датчика). Обработку производит OrangePi.

### 5.3. Команда на действие (Command Type: 10)
Поле DATA длиной 1 байт содержит код команды. OrangePi отправляет команду, STM32 исполняет её немедленно.
*   `1` — Старт вибромоторов.
*   `2` — Стоп вибромоторов.
*   `3` — Старт нагрева воздуха.
*   `4` — Стоп нагрева воздуха.
*   `5` — Старт нагрева крыши.
*   `6` — Стоп нагрева крыши.
*   `7` — Старт светодиодной ленты.
*   `8` — Стоп светодиодной ленты.
*   `9` — Открыть вентиляцию.
*   `10` — Закрыть вентиляцию.
*   `11` — Начать вентилировать воздух.
*   `12` — Остановить вентилировать воздух.
*   `13` — Поднять стол.
*   `14` — Опустить стол.
*   `15` — Открыть зарядные лапки дрона.
*   `16` — Закрыть зарядные лапки дрона.
*   `17` — Открыть крышу дронпорта.
*   `18` — Закрыть крышу дронпорта.
*   `19` — Включить климат-контроль.
*   `20` — Выключить климат-контроль.

### 5.4. Сообщение об ошибке (Command Type: 11)
Поле DATA длиной 1 байт содержит код ошибки, зависящий от поля System Part.
*   **Датчики Холла (001):** Номер датчика, от которого не поступил сигнал.
*   **Напряжение (010):** `1` — напряжение слишком низкое, `2` — напряжение слишком высокое.
*   **DHT22 (011):** `0` — слишком жарко, `1` — слишком холодно, `2` — слишком влажно, `3` — слишком сухо, `4` — ошибка контрольной суммы датчика.
*   **Программная часть (000):** `0` — ошибка контрольной суммы пакета, `1` — переполнение буфера пакета, `2` — переполнение буфера пакетов.

## 6. Контрольная сумма (Checksum)
Целостность пакета обеспечивается байтом контрольной суммы. Алгоритм расчета следующий:
1.  Суммируются значения всех байт пакета, начиная от `BYTE_START` и заканчивая `BYTE_END`, за исключением самого байта `CHECKSUM`.
2.  Из полученной суммы берется младший байт (операция по модулю 256).
3.  Формула: `CHECKSUM = (BYTE_START + TYPE + LEN + DATA_BYTES + BYTE_END) % 256`.

## 7. Тайминги и логика обмена
### 7.1. Межбайтовый таймаут
Приемник должен отслеживать время поступления байтов. Если интервал между двумя соседними байтами одного пакета превышает **100 мс**, приемник обязан считать пакет невалидным, сбросить буфер приема и ожидать новый байт `BYTE_START`.

### 7.2. Ограничение на содержимое данных (Data Integrity Constraint)
Механизм экранирования (byte stuffing) не используется. Значения байтов `0xBF` и `0xFF` зарезервированы под маркеры начала и конца пакета.
*   **Требование:** При формировании поля DATA (особенно для напряжения и сырых данных DHT22) отправитель обязан гарантировать, что полезные данные не совпадают с маркерами. Допускается незначительная модификация передаваемых данных на уровне приложения для соблюдения этого правила, если возникает коллизия.

### 7.3. Подтверждение действий (Action Acknowledgement)
Протокол не предусматривает явного подтверждения выполнения команды действия (Action Command).
*   **Логика OrangePi:** После отправки команды действия, OrangePi ожидает настраиваемый таймаут (зависит от команды, конфигируется в `hardware.yaml`).
*   **Верификация:** По истечении таймаута OrangePi отправляет Запрос Статуса (например, запрос датчиков Холла для проверки положения механизма).
*   **Повторные попытки:** Если статус не соответствует ожидаемому, OrangePi повторяет цикл «Таймаут -> Запрос статуса» суммарно до 3 раз.
*   **Ошибка механики:** Если после 3 попыток статус не изменился, OrangePi фиксирует ошибку механики и отправляет отчет на удаленный сервер.

## 8. Обработка ошибок
### 8.1. Ошибка контрольной суммы на стороне STM32
Если STM32 принимает пакет с невалидной контрольной суммой:
1.  Пакет игнорируется.
2.  STM32 немедленно формирует и отправляет обратно пакет типа «Ошибка» (Command Type: 11, System Part: 000, Data: 0).

### 8.2. Ошибка контрольной суммы на стороне OrangePi
Если OrangePi принимает пакет с невалидной контрольной суммой:
1.  Пакет игнорируется и логируется локально.
2.  Если ожидался ответ на запрос, OrangePi повторяет отправку исходного запроса.
3.  **Лимит повторений:** Допускается максимум 3 повторных запроса подряд.
4.  **Критический сбой:** Если после 3 попыток контрольная сумма снова не сходится, OrangePi прекращает попытки и отправляет сообщение об ошибке связи на удаленный сервер.

### 8.3. Ошибка выполнения команды
Если STM32 не может выполнить команду действия мгновенно (например, механическое препятствие), он должен немедленно отправить пакет типа «Ошибка» с соответствующим кодом в поле DATA, не дожидаясь опроса со стороны OrangePi.

## 9. Инициализация системы
Система не требует процедуры рукопожатия. При подаче питания:
1.  OrangePi и STM32 инициализируют свои периферийные модули (UART, GPIO, датчики) независимо.
2.  После завершения внутренней инициализации устройства переходят в режим ожидания входящих пакетов.
3.  OrangePi может инициировать первый диагностический запрос статуса сразу после загрузки своей операционной системы для проверки связи.
