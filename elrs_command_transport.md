# Передача по ELRS: команды (станция→дрон) и данные (дрон→станция)

Радиоканал — HappyModel ES900 (ELRS, 900 МГц). UART 420000 8N1, half-duplex.
Всё — CRSF. Кадр CRSF: `ADDR | LEN | TYPE | PAYLOAD | CRC8(0xD5)`, `LEN = len(PAYLOAD)+2`,
CRC8 по `TYPE+PAYLOAD`. Адреса: дрон `0xC8`, станция `0xEE`, radio (LINK_STATS) `0xEA`.

## Главное ограничение ELRS
ELRS прозрачно передаёт **только стандартные кадры**, и не все одинаково:
- **RC_CHANNELS_PACKED (0x16)** — 16 каналов; но честно (полным разрешением, каждый OTA-кадр)
  доходят только **CH1–CH4**. CH5–CH16 (aux) режутся до 6–7 бит и шлются по кругу (round-robin) —
  под данные непригодны.
- **Стандартная телеметрия** (GPS 0x02, ATTITUDE 0x1E, BATTERY 0x08, LINK_STATS 0x14) — релеится.
- **Кастомные CRSF-типы** (0x11–0x2E, 0xF1/0xF2) по воздуху **НЕ проходят**.

Отсюда обе схемы ниже: команды прячутся в каналы RC_CHANNELS, данные дрона — в стандартные
телеметрийные кадры.

---

## 1. Станция → дрон: КОМАНДЫ (поверх RC_CHANNELS, фрагментация)

Каждый кадр — стандартный `RC_CHANNELS_PACKED` (ADDR=0xC8, TYPE=0x16, PAYLOAD 22 байта = 16×11 бит),
темп 100 Гц. Данные кладутся только в надёжные **CH1–CH4**, CH5–CH16 = нейтраль (992).

### Кодирование значения в канал (stride-2)
ELRS теряет младший бит (даунсемпл 11→10). Поэтому:
```
raw = 992 + value*2      (value = 0..255  →  raw = 992..1502)
value = round((raw - 992) / 2)
```
Чётная база 992 + шаг 2 ⇒ младший бит не несёт информации, значение выживает.

### Раскладка одного фрагмента
| Канал | Значение (до stride-2) | Назначение |
|---|---|---|
| CH1 | `cmd_id` | тип команды (0x11…0x1B); в каждом фрагменте |
| CH2 | `(session<<4) \| frag_index` | сессия (4 бита) + индекс фрагмента (4 бита) |
| CH3 | `data[2*frag_index]` | байт данных |
| CH4 | `data[2*frag_index+1]` | байт данных (последний нечётный → 0) |
| CH5–CH16 | 992 | нейтраль |

`cmd_id = 0` (CH1=992) ⇒ кадр-**beacon** (команды нет, держит линк).

### Полезная нагрузка и фрагменты
- Передаётся `payload + CRC8(payload)` (1 байт), нарезается по **2 байта на кадр**.
- Число фрагментов: `total = ceil((len(payload)+1)/2)`. Получатель берёт `len(payload)` из `cmd_id`
  (фиксированный размер каждой команды), поэтому длину в эфире не шлём.
- Станция повторяет весь набор фрагментов `repeat_cycles` раз (деф. 3) — устойчивость к потерям.
- Дрон собирает по `session`, проверяет CRC, обрабатывает один раз (дедуп по session).
- Новая команда/NED увеличивает `session` и заменяет программу (newest-wins).

### Размеры команд
| Команда | cmd_id | payload, байт | фрагментов |
|---|---|---|---|
| STOP / RETURN / PRE_FLIGHT / DIAGNOSTIC_FLIGHT | 0x1A/0x1B/0x15/0x16 | 0 | 1 |
| OPEN_DRONEPORT | 0x18 | 1 | 1 |
| COMBAT_MODE / TARGET_INTERCEPTION / DEMO_MODE / COORDINATE_NED | 0x11/0x12/0x17/0x19 | 13 | 7 |
| DRONE_FLIGHT | 0x14 | 12 | 7 |
| SECTOR_SEARCH | 0x13 | 16 | 9 |

### Пример (COMBAT_MODE x=10,y=20,z=30)
payload(13Б) = `0A000000 14000000 1E000000 00`; +CRC8 = 14 байт → 7 фрагментов.
Кадр фрагмента 0: CH1=enc(0x11), CH2=enc(session<<4|0), CH3=enc(0x0A), CH4=enc(0x00), CH5–16=992.

### Темп и латентность
- Кадры 100 Гц. Доставка команды ≈ `total × repeat_cycles × 10 мс` (COMBAT ≈ 7×3×10 = 210 мс).
- **COORDINATE_NED** (поток): эффективная частота ≈ `100Гц / total` ≈ 14 Гц (полная точность важнее
  частоты — согласовано). Каждый новый NED = новая session; дрон отдаёт последний полностью собранный.

## 2. Дрон → станция: ДАННЫЕ (стандартные телеметрийные кадры)

Кастомные типы по эфиру не проходят, поэтому вся телеметрия разложена по **стандартным CRSF-кадрам**
(ADDR=0xC8), которые ELRS релеит. Темп ~10 Гц.

| Данные | TYPE | Формат (LE) | Содержимое |
|---|---|---|---|
| GPS | 0x02 | `<iiHHHB>` (15Б) | lat×1e7, lon×1e7, NED X/Y/Z (i16, м), battery% |
| ATTITUDE | 0x1E | `<hhh>` (6Б) | roll/pitch/yaw ×100° |
| BATTERY | 0x08 | `<HH>+[flags,mode,event]+[batt%]` (8Б) | current×10, speed×100, флаги, mission_mode, **event-байт**, battery% |
| LINK_STATS | 0x14 | (от ES900RX, ADDR=0xEA) | RSSI/LQ восходящего канала |

### События дрона (SOS / TARGET / BOARDING / RETURN) — в event-байт кадра 0x08
Поскольку отдельные кадры событий ELRS не пропускает, событие латчится в 3-й байт mah кадра BATTERY:
| Биты | Поле | Значения |
|---|---|---|
| 0–2 | id | TARGET=1, BOARDING_REQUEST=2, SOS=3, RETURN_DRONE=4 |
| 3–5 | value | 0..7 |
| 6 | toggle | инверсия при каждом новом событии (дедуп) |

Повтор события несколько кадров (`event_repeat_frames`), станция распознаёт по `toggle`.

### Подтверждения команд (ACK) — оговорка
Кастомный кадр ACK `0xF1` по реальному эфиру ELRS **не релеится** (как и любой нестандартный тип).
- В диагностическом скрипте `rc_link_test.py` ACK `0xF1` работает, т.к. стенд — прямой/half-duplex UART.
- Для подтверждения команд **по воздуху** ACK нужно piggyback-ить в стандартный кадр (например,
  отдельное поле «последний принятый cmd_id+session» в event-байте/резерве кадра 0x08), а не слать
  отдельным 0xF1. Это TODO для over-air ACK.
